# 儿童考试系统 · 服务器部署与一键升级

> 配套脚本：`deploy.sh`（支持 init / upgrade / backup / rollback / status / logs / restart / stop / start）

## 一、服务器前置条件（只需三项）

- **Docker**（含 `docker compose` 子命令，建议 20.10+）
- **git**
- 能访问 GitHub（拉代码）与 Docker Hub（拉 `busybox` 做数据库备份）

> 不需要在服务器装 Node / 编译工具——所有构建都在镜像构建阶段完成。

## 二、首次部署（一键）

```bash
git clone git@github.com:xueliangGit/homework.git
cd homework
./deploy.sh init
```

脚本会自动：
1. 生成 `.env`（随机 `JWT_SECRET`，避免用默认弱密钥）；
2. 构建镜像并启动容器（首次需编译 `better-sqlite3`，约 1–3 分钟）；
3. 等待健康检查通过后打印访问地址 `http://localhost:4000`。

数据库落在命名卷 `kaoshi_data`，容器重建 / 重启 / 服务器重启（`restart: unless-stopped`）都不丢数据。

## 三、以后升级（一键）

```bash
./deploy.sh upgrade
```

脚本会自动：
1. 备份数据库卷到 `backups/kaoshi-db-<时间>.tar.gz`；
2. `git pull` 拉取最新代码；
3. 重建镜像并重启容器；
4. 健康检查；**若新版本启动失败，自动回滚到升级前版本 + 恢复数据库**。

升级失败兜底：
```bash
./deploy.sh rollback    # 回滚到上一次 upgrade 前的版本与数据库
./deploy.sh backup      # 随时手动备份
```

## 四、常用运维

```bash
./deploy.sh status   # 容器 / 镜像 / 卷状态
./deploy.sh logs     # 跟踪日志
./deploy.sh restart  # 重启（改了 .env 后用它生效）
./deploy.sh stop     # 停止
./deploy.sh start    # 启动
```

## 五、上生产的两件必做事项

1. **改密钥 / 跨域**：编辑 `.env` 的 `JWT_SECRET`（已是随机）与 `CORS_ORIGIN`，改完 `./deploy.sh restart`。
2. **HTTPS + 域名（强烈建议）**：默认只暴露 `4000` 端口。公网部署请在前端加反向代理做 TLS，例如 Caddy：

   ```Caddyfile
   your.domain.com {
     reverse_proxy localhost:4000
   }
   ```

   用 `caddy run` 或 `docker run -d -p 80:80 -p 443:443 -v $PWD/Caddyfile:/etc/caddy/Caddyfile caddy` 即可自动签发证书。

## 六、想升级更快？（可选：预构建镜像）

`upgrade` 每次都在服务器本地重新编译镜像（含原生模块），较慢。若要秒级升级：

1. 把镜像推到容器 registry（如 GitHub 的 GHCR）；
2. 把 `docker-compose.yml` 的 `build:` 段去掉、改为 `image: ghcr.io/你的账号/kaoshi:latest`；
3. 升级时执行 `docker compose pull && docker compose up -d`。

（当前默认走"源码构建"路径，好处是零额外配置，clone 即可用。）

## 七、已知约定与坑

- SQLite 使用 **WAL 模式**，数据目录 `/app/.data` 已由卷持久化。
- 数据库表用 `CREATE TABLE IF NOT EXISTS` 初始化，结构向后兼容，升级一般无需手工迁移。
- 备份/恢复使用 `busybox` 临时容器挂载同一卷，不依赖宿主机 tar 版本。
- 升级脚本通过容器的 `Mounts` 反查真实卷名（compose 会给卷加项目前缀），因此不受目录名影响。
